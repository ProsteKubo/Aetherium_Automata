# Refocus Plan for Aetherium: From Wide Prototype to Coherent “Behemoth”

## Summary

The repo already has a real core, but the project’s center of gravity is wrong.

Current tracked code is roughly:
- `src/ide`: 51k lines
- `src/server`: 17k lines
- `src/engine`: 15k lines
- `src/gateway`: 10k lines

That means the largest investment is in UI and orchestration breadth, not in the runtime semantics that would make Aetherium unique. The immediate goal should be to **shrink the active surface area** and make one story executable end-to-end:

**black-box distributed automata + deployment-aware execution + fault-in-the-loop validation + replayable traces**

## Current State Analysis

- **What is solid already**
  - Engine core exists: parser, runtime, protocol codec, CLI runner, embedded targets.
  - Server already owns the real orchestration value: device sessions, deploy flow, target profiles, runtime state, time-series groundwork.
  - Protocol and deploy pipeline are real enough to support a serious thesis direction.
  - Examples and showcase material are useful for demos and validation.

- **What is slowing the project down**
  - IDE is the biggest tracked subsystem and contains a lot of breadth before the product semantics are settled.
  - Several large IDE panels are mock/demo-heavy or partially placeholder-backed, especially topology and time-travel views.
  - Gateway and server are both evolving as control-plane layers, which creates split-brain architecture pressure.
  - Multiple transports and device families are pursued at once: WS, serial, ROS2; AVR, ESP32, MCXN947.
  - Observability stack has already spread into local time-series, Influx, Grafana, replay, dashboards before fault/deployment semantics are first-class.
  - `DeviceManager` is carrying too many responsibilities at once, which is a sign the domain model is not separated cleanly enough yet.

- **What is not necessary right now**
  - Polished multi-panel Electron workbench as the primary development focus.
  - ROS2 as a first-class thesis path.
  - Multiple MCU families in active development.
  - Grafana/Influx as a required path for core validation.
  - A rich gateway/server split unless there is a hard deployment requirement for it now.
  - Broad “network command center” UI before the underlying metrics are generated by the real runtime.

## Implementation Changes

### 1. Pick the canonical product slice and freeze the rest
- Active scope for the next major milestone:
  - host runtime
  - one real embedded target: `ESP32`
  - one primary control service: `server`
  - one primary device path: `serial` or `WebSocket`
  - one trace store path: local built-in store
- Freeze, hide, or mark secondary:
  - ROS2 bridge
  - MCXN947
  - AVR beyond smoke support
  - Grafana/Influx as optional adapters
  - mock-heavy IDE panels not backed by live server data

### 2. Make the domain model explicit
- Introduce three first-class concepts and keep them separate:
  - **Automaton model**: states, transitions, ports, local variables, stochastic behavior.
  - **Deployment descriptor**: which instance runs where and through what transport.
  - **Fault profile**: which boundaries can delay, drop, fail, or distort behavior.
- Do not bury deployment and failure semantics inside ad hoc server logic or UI-only concepts.
- Black-box contracts must explicitly define:
  - input ports
  - output ports
  - emitted events
  - observable state hooks
  - allowed fault injection points

### 3. Re-architect around the server as the system brain
- Recommended default: the server becomes the main orchestrator and source of truth.
- Gateway should be reduced to one of two roles:
  - thin authenticated ingress/relay only, or
  - temporarily dormant while the server-centric thesis slice is built
- Split `DeviceManager` responsibilities into focused services:
  - device/session registry
  - deployment coordinator
  - runtime state projection
  - fault-injection coordinator
  - trace/replay coordinator
- Unify on protocol v2 as the real protocol path; keep v1 as compatibility documentation only.

### 4. Build the unique part next: deployment-aware runtime + fault-in-the-loop
- Add deployment metadata to runtime events:
  - source instance
  - target instance
  - transport type
  - placement
  - timestamps for send/receive/handle
- Implement fault injectors as explicit runtime/deployment components, not hidden hacks:
  - delay
  - jitter
  - drop
  - duplicate
  - degraded success probability
  - intermittent disconnect
- Validation profiles enable injectors.
- Production profiles disable injectors without changing automata logic.

### 5. Narrow the UI to an operator workbench, not a universe
- Keep only real-data-backed UI surfaces in the active roadmap:
  - automata editor
  - deploy/start/stop control
  - runtime state monitor
  - event trace / replay inspector
  - minimal topology/deployment view
- Remove from active development anything that is still mostly mocked or decorative.
- Replace giant all-in-one panels with smaller views bound directly to server APIs and trace data.
- Time-travel and network views should not evolve further until they read actual trace/deployment data end-to-end.

### 6. Add Petri-net analysis only as a derived analyzer
- Do not build a second runtime or second modeling environment.
- Build a projection pipeline from execution traces / deployment graph to a contention view.
- First supported analyzer question:
  - “Which automata/processes compete for the same resource or queue?”
- Inputs to the analyzer:
  - declared shared resources
  - deployment graph
  - event/transition traces
  - fault-injection outcomes
- Output:
  - blocked paths
  - token accumulation
  - starvation/contention hot spots

### 7. Define milestone order for the “behemoth”
- **Milestone 1: Core narrowing**
  - freeze scope
  - choose canonical target stack
  - server-centered architecture cleanup
- **Milestone 2: Domain separation**
  - automaton model vs deployment descriptor vs fault profile
  - black-box contracts
- **Milestone 3: Fault-in-the-loop runtime**
  - deployable injectors
  - timestamped event metadata
  - trace capture
- **Milestone 4: Replay and measurement**
  - deterministic replay where possible
  - end-to-end delay/jitter instrumentation
  - placement comparison experiments
- **Milestone 5: Petri-style analyzer**
  - resource/contention projection
  - system-level analysis view
- **Milestone 6: UI consolidation**
  - rebuild only the views needed to demonstrate the above coherently

## Test Plan

- **Core runtime**
  - load and run one host automaton with stochastic/fault-capable transitions
  - verify explicit boundary instrumentation and trace capture

- **Distributed path**
  - deploy one automaton locally and one on ESP32
  - verify communication over the chosen primary transport
  - record send/receive/handle timestamps

- **Fault-in-the-loop**
  - delay injection
  - drop injection
  - degraded-success injection
  - verify recovery paths appear in replay exactly as injected

- **Deployment-aware experiments**
  - same-host run
  - host-to-device run
  - remote/gateway-like delayed path
  - compare median, p95, p99 latency and behavioral impact

- **Analyzer**
  - one shared-resource scenario with visible contention
  - verify Petri-style projection highlights rivalry/blocking from real traces

- **UI acceptance**
  - every active panel must be backed by real runtime/server data
  - no active thesis-critical panel may depend on mock-generated metrics or logs

## Assumptions and Defaults

- The project should optimize for **coherence over breadth** for the next major cycle.
- The correct thesis/product center is the **runtime and validation semantics**, not the size of the IDE.
- The server is assumed to be the primary orchestrator unless a hard requirement forces a full gateway-first architecture.
- ESP32 is the primary embedded target; other hardware paths are secondary until the core story is proven.
- Petri nets are assumed to be an analyzer layer, not a second execution formalism.
- Grafana, ROS2, and extra targets remain valuable, but only after the core deployment-aware fault-validation loop is complete.
